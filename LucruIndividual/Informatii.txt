=======================================
         DESIGN PATTERNS CHEAT SHEET
=======================================

--- SINGLETON (Creational) ---
3 lucruri obligatorii:
1. private static Singleton instanta = null;
2. private Singleton(...)  — constructor PRIVAT
3. public static synchronized Singleton getInstance(...)
   -> daca instanta == null, o creaza; altfel returneaza aceeasi

Nu folosim: clasa abstracta (nu poate fi instantiata), interfata (nu controleaza instantierea)
Cuvinte cheie in enunt: "o singura instanta", "un singur sistem", "acelasi obiect"

--- SIMPLE FACTORY (Creational) ---
4 actori:
1. <<Interface>> Produs       — interfata primita (ex: Bautura, Floare)
2. Clase concrete              — implementeaza interfata (ex: Cafea, Ceai, CiocolataCalda)
3. Factory                     — clasa cu metoda STATIC care creeaza obiectele
4. Client (Main)               — foloseste DOAR interfata + fabrica, NU clasele concrete

Fabrica:
- metoda: public static InterfataProdus creeaza(TipEnum tip, argumente...)
- returnează INTERFATA, nu clasa concreta (DIP!)
- switch pe enum -> new ClasaConcreta(...)
- default -> throw new IllegalArgumentException(...)

Main-ul e ORB:
- NU importa clase concrete (Cafea, Ceai etc.)
- importa DOAR: interfata + fabrica + enum
- variabile de tip INTERFATA: Bautura b = FabricaBauturi.getBautura(...)

Enum pentru tipuri:
- nu String, ca sa nu gresesti la scriere
- IDE-ul da autocomplete

Cuvinte cheie in enunt: "crearea separata de logica principala", "diferite tipuri de..."

--- FACTORY METHOD (Creational) ---
Diferenta fata de Simple Factory:
- Simple Factory = 1 fabrica, 1 familie de produse
- Factory Method = MAI MULTE fabrici, MAI MULTE familii de produse
- Fabrica insasi e polimorfica (interfata + clase concrete)

Structura (6 actori):
1. <<Interface>> Produs          — interfata comuna (ex: FelDeMancare, PersonalSpital)
2. Clase abstracte per familie   — cate una per categorie (ex: Supa, Desert)
   -> atribute DIFERITE per familie (de asta exista clase abstracte!)
3. Clase concrete                — extends clasa abstracta (ex: SupaDeLegume extends Supa)
4. <<Interface>> Fabrica         — interfata fabricii cu metoda fabrica
5. Fabrici concrete              — cate una per familie (ex: FabricaSupa, FabricaDesert)
6. Client (Main)                 — dublu DIP: interfata fabrica + interfata produs

Enum-uri (trick-ul):
- TipFelDeMancare (interfata GOALA) — lipiciul comun
- TipSupa implements TipFelDeMancare
- TipDesert implements TipFelDeMancare
- Asa metoda fabricii accepta ambele enum-uri prin acelasi parametru

Interfata fabricii — overload-uri:
- creare(tip, gramaj, pret)           // pentru familia fara calorii
- creare(tip, gramaj, pret, calorii)  // pentru familia cu calorii
- Fabrica care nu foloseste un overload DELEGHEAZA catre celalalt (nu return null!)

Main — dublu DIP:
- FabricaFelDeMancare fabrica = new FabricaSupa();    // DIP pe fabrica
- FelDeMancare supa = fabrica.creare(...);             // DIP pe produs
- NU importa clase concrete, DOAR interfete + fabrici + enum-uri

Clase abstracte — CAND le folosim:
- DOAR cand familiile au ATRIBUTE DIFERITE (Supa: gramaj,pret vs Desert: gramaj,pret,calorii)
- La Simple Factory NU le folosim (toate clasele au aceleasi atribute)
- toString() se pune in clasa abstracta, afisareDescriere() in clasa concreta

Cuvinte cheie in enunt: "familie de obiecte impartite pe CATEGORII", "doua tipuri de..."
